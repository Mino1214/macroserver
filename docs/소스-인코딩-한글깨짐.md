# 한글이 `??`로 보일 때 (서버.js 등)

## 결론: **Git 커밋이 인코딩을 망가뜨리지 않습니다**

`git commit` / `git push`는 **이미 디스크에 있는 바이트**를 그대로 기록합니다.  
커밋만 했다고 해서 UTF-8 한글이 `?`로 바뀌지는 않습니다.

## 실제로 무슨 일이 있었는지

`server.js` 안의 한글은 **“잘못 읽힌 UTF-8”이 아니라, 글자가 통째로 사라지고 물음표로 대체된 상태**입니다.

- `file -bi server.js` → `charset=us-ascii` 처럼 나오는 이유: **한글 UTF-8 바이트가 없고**, `?`(ASCII)만 남아 있기 때문입니다.
- 즉 **에디터/도구가 한글을 저장할 때 이미 `?`로 덮어썼거나**, 다른 인코딩으로 연 뒤 저장한 적이 있는 겁니다.

## 이 저장소에서 깨지기 시작한 커밋 (확인됨)

히스토리를 따라가 보면:

- **`47f1418` 이전** (`47f1418^`): `// MariaDB 연결` 처럼 **한글 정상**
- **`47f1418`부터**: `// MariaDB ??` 처럼 **한글 깨짐**

커밋 메시지: `feat: telegram channels - master deposit/seed/approval, ...`

즉, **최근 APK 수정 커밋(`80e5e81`) 때문이 아니라**, 그보다 이전에 **`47f1418`에서 이미 파일이 깨진 상태로 들어간 것**입니다.

## 앞으로 막는 방법

1. **에디터에서 `server.js` 인코딩을 UTF-8로 고정** (BOM 없음 권장).
2. 프로젝트 루트의 **`.editorconfig`** 를 켜 두기 (이미 추가됨 → VS Code / Cursor 등에서 EditorConfig 확장 사용).
3. 큰 패치를 **다른 OS/도구에서 붙여넣기** 할 때, 중간에 “ANSI만 지원” 창을 거치지 않기.

## 복구하고 싶다면

- **한글 주석/문구만** 예전 버전에서 가져오기:  
  `git show 47f1418^:server.js` 와 현재 `server.js`를 **diff** 해서 필요한 부분만 수동 반영.
- **전체를 예전으로 되돌리기**는 기능이 많이 달라져서 비추천 (충돌·기능 유실).

## API JSON이 깨져 보이는 경우

`res.json({ error: '...' })` 안의 문자열이 소스에 이미 `?`이면, 응답도 `?`만 갑니다.  
**소스 UTF-8 복구** 또는 **에러 메시지는 영문 + 코드**로 통일하는 방식이 안전합니다.
